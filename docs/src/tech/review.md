# Clotho App 视图设计技术评审报告

## 1. 评审总结

### 1.1 总体评价

五个视图的详细设计文档整体质量较高，交互规格清晰，数据模型引用一致。技术选型合理，基于 Tauri 2 + React 19 + shadcn/ui 的方案能够满足大部分设计需求。

**主要结论**：
- 项目列表视图和列表视图可直接进入开发，设计成熟度高
- 看板视图设计完善，dnd-kit 方案可行
- 甘特图视图是最大技术风险点，自研 SVG 方案工作量大且难度高
- 日历视图依赖 Schedule-X，集成风险中等

### 1.2 整体难度评级

| 视图 | 难度 | 预估工作量（人天） | 风险等级 |
|------|------|-------------------|---------|
| 项目列表 | 简单 | 5-7 天 | 低 |
| 列表视图 | 中等 | 8-10 天 | 低 |
| 看板视图 | 中等 | 10-12 天 | 中 |
| 日历视图 | 中等偏难 | 10-14 天 | 中 |
| 甘特图视图 | 困难 | 18-25 天 | 高 |

**总计**：51-68 人天（约 2.5-3.5 个月单人开发）

## 2. 各视图评审详情

### 2.1 项目列表视图

**难度**：简单

**技术可行性**：完全可行。所有设计均在技术选型范围内。

**优点**：
- 设计精细，行模式和卡片模式的切换逻辑清晰
- 展开/折叠的二级任务列表是不错的创新点
- 数据需求接口定义清晰（ProjectListItem、TaskListItem）
- 性能策略合理：项目数量通常 <100，无需虚拟滚动

**问题与建议**：

1. **进度计算位置**：文档提到"进度计算在 Rust 侧完成"，建议在 `list_projects` 的 Tauri command 中直接用 SQL 聚合完成（`COUNT` + `CASE WHEN status = 'done'`），而非在 Rust 代码中手动计算。

2. **展开任务列表的虚拟滚动**：文档提到"超过 50 条使用 TanStack Virtual"。但展开状态下任务列表嵌在项目行之间，虚拟滚动的容器高度计算会比较复杂。建议 MVP 阶段简化为：展开时最多显示 20 条任务 + "查看全部 N 条"链接（点击跳转到项目的列表视图）。

3. **卡片模式拖拽排序**：文档提到"长按 200ms 或拖拽手柄触发"，但卡片模式的视觉设计中没有明确拖拽手柄的位置。建议与行模式一致，悬停时显示 `GripVertical` 图标。

4. **搜索匹配方式**：文档说"不区分大小写的子字符串匹配"。SQLite 的 `LIKE` 对中文是大小写敏感的，需要使用 `COLLATE NOCASE` 或在 Rust 侧做 `LOWER()` 处理。

### 2.2 看板视图

**难度**：中等

**技术可行性**：可行。dnd-kit 是成熟方案，closestCorners 碰撞检测适合看板场景。

**优点**：
- dnd-kit 配置细节完善（sensor、collision detection、DragOverlay）
- 拖拽视觉反馈规格清晰（半透明、阴影、旋转）
- WIP 限制是好功能，实现简单但价值高
- 乐观更新 + 回滚策略合理

**问题与建议**：

1. **筛选策略的性能隐患**：文档设计"不匹配的卡片不隐藏，而是 opacity: 0.3"。当任务量大时（500+ 张卡片），所有卡片仍然在 DOM 中渲染。建议在任务数量超过阈值（如 200）时改为隐藏不匹配的卡片，或者结合虚拟滚动只渲染可见区域的卡片。

2. **按标签分组时列数过多**：如果用户创建了 20+ 个标签，按标签分组会产生 20+ 列，水平滚动体验不佳。建议限制最多显示 10 列，超出部分合并到"其他"列，或增加标签筛选后再分组。

3. **kanban_order 排序算法**：文档提到"使用间隔算法避免频繁重排"。建议采用 fractional indexing（分数排序）方案，例如使用字符串排序键（"a0", "a0V", "a1"），避免整数排序需要批量更新的问题。可参考 `fractional-indexing` npm 包。

4. **折叠列的拖拽行为**：文档未明确：卡片能否拖拽到折叠列中？建议允许拖拽到折叠列（折叠列在拖拽 hover 时短暂展开预览），否则用户需要先展开列才能操作，体验不佳。

5. **触摸设备的 `navigator.vibrate`**：Tauri 桌面应用中 `navigator.vibrate` 不可用（仅移动浏览器支持）。建议移除此特性或改用系统原生触觉反馈。

### 2.3 列表视图

**难度**：中等

**技术可行性**：完全可行。TanStack Table + TanStack Virtual 是成熟方案。

**优点**：
- TanStack Table 的 headless 特性与 shadcn/ui 完美配合
- 虚拟滚动配置合理（estimateSize: 40px, overscan: 5）
- 行内编辑的 Tab 导航设计流畅
- 批量操作栏设计完善

**问题与建议**：

1. **列固定与虚拟滚动的兼容**：文档要求"复选框列和任务名称列固定在左侧"。TanStack Table 的列固定 + TanStack Virtual 的虚拟滚动同时工作时，需要注意 CSS 的 `position: sticky` 在虚拟滚动容器中的表现。建议提前做技术验证 PoC。

2. **分组头拖拽排序**：文档说"分组头可拖拽调整顺序（拖拽时整组移动）"。在虚拟滚动场景下，拖拽整个分组（可能包含数百行）的视觉反馈和性能是个挑战。建议 MVP 阶段不支持分组头拖拽，改为下拉菜单调整分组顺序。

3. **筛选面板的"在...之间"操作符**：截止日期筛选支持"在...之间"需要两个日期输入。建议使用 shadcn/ui 的 DateRangePicker 组件（需要确认是否已有此组件，否则需要基于 DatePicker 自行组合）。

4. **行内编辑状态下的虚拟滚动**：如果用户正在编辑某个单元格，此时滚动导致该行被虚拟化（移出可视区域再回来），编辑状态会丢失。建议编辑状态下临时禁用该行的虚拟化，或在行离开可视区域时自动保存。

### 2.4 甘特图视图

**难度**：困难

**技术可行性**：可行但风险高。自研 SVG 甘特图是整个项目最大的技术挑战。

**优点**：
- 设计文档非常详细，SVG 结构定义清晰
- 依赖关系连线的贝塞尔曲线方案合理
- 关键路径计算放在 Rust 侧是正确的架构决策
- 拖拽吸附到日边界的交互设计符合用户预期

**问题与建议**：

1. **SVG 虚拟滚动的复杂度**（高风险）：文档提到"超过 100 个可见任务行时启用 TanStack Virtual，SVG 元素动态添加/移除"。但 TanStack Virtual 设计用于 DOM 列表虚拟化，而非 SVG 元素。在 SVG 中实现虚拟化需要：
   - 手动管理 SVG 元素的创建和销毁
   - 正确计算 SVG viewBox 和 transform
   - 处理依赖连线跨越可视区域边界的情况（连线两端可能一个在可视区内一个在区外）

   **建议**：将 TanStack Virtual 用于左侧任务列表面板（HTML 列表），右侧 SVG 甘特图使用自定义的基于 `scrollTop` 的渲染裁剪。对于依赖连线，始终渲染所有连线（线条的 SVG 开销远小于复杂图形）。

2. **左右面板同步滚动的抖动**（中风险）：文档提到"监听两侧 scroll 事件，同步 scrollTop 值"。双向 scroll 同步容易产生循环触发和抖动。

   **建议**：使用单一滚动容器方案 — 左右面板都不独立滚动，而是放在同一个外层滚动容器内，仅右侧甘特图有独立的水平滚动。或者使用 `overflow: hidden` + `wheel` 事件手动控制滚动位置。

3. **SVG 拖拽与 dnd-kit 不兼容**：文档正确指出甘特图拖拽使用原生 pointer 事件而非 dnd-kit。但这意味着甘特图的拖拽系统是独立实现的，无法复用看板的拖拽代码。需要额外的开发和测试工作量。

4. **依赖关系连线的交互热区**：贝塞尔曲线的点击/悬停检测比较困难（SVG path 的 stroke-width 1.5px 太窄）。建议为每条连线添加一个透明的宽热区 path（stroke-width: 10px, stroke: transparent），用于接收鼠标事件。

5. **性能目标过于激进**：
   - "首次渲染 < 200ms（100 个任务条）"：合理
   - "关键路径计算 < 100ms（200 个任务 + 100 条依赖）"：如果放在 Rust 侧，CPM 算法复杂度 O(V+E)，200 个任务是没问题的
   - "依赖连线渲染 < 50ms（50 条依赖）"：贝塞尔曲线计算 + SVG path 渲染 50 条，需注意避免触发大量 reflow

6. **仅有截止日期无开始日期的任务**：文档说"开始日期默认为截止日期前 1 天"。这会导致很多只设了 due_date 的任务在甘特图上显示为 1 天的窄条，信息密度低。建议改为：无 start_date 的任务在甘特图上显示为菱形标记（类似里程碑），鼠标悬停提示"未设置开始日期"。

7. **缩放过渡动画**：文档说"缩放以鼠标光标所在时间点为中心，过渡动画 200ms"。SVG 缩放动画在大量元素时可能卡顿。建议缩放时先降低渲染精度（隐藏文字标签、简化任务条），动画完成后恢复完整渲染。

### 2.5 日历视图

**难度**：中等偏难

**技术可行性**：可行，但 Schedule-X 集成存在定制化上限的风险。

**优点**：
- Schedule-X 是轻量现代的日历库，与 shadcn 风格一致
- CSS 变量主题定制方案清晰
- 迷你日历导航是好的 UX 设计
- 月/周/日三种子视图的数据映射规则清晰

**问题与建议**：

1. **Schedule-X 定制深度**（中风险）：文档计划使用 Schedule-X 的 custom event renderer 来自定义事件样式。但 Schedule-X 的以下功能可能需要验证：
   - 月视图中跨天事件的渲染是否支持自定义（Schedule-X 的跨天事件布局算法可能是内置的）
   - "+N more" 的样式和交互是否可定制
   - 拖拽时的 tooltip 是否可以自定义内容

   **建议**：在开发前用 1-2 天时间做 Schedule-X 的定制化验证 PoC，确认以上功能的可定制程度。如果定制受限，需要 fork 或切换到 FullCalendar。

2. **数据模型不匹配**：日历视图设计中提到了"日程事件（Calendar Event）"和"时间段"概念，但当前数据模型中 Task 只有 `start_date`（DATE 类型）和 `due_date`（DATE 类型），没有具体的时间字段（如 `start_time`、`end_time`）。

   周/日视图中需要显示精确到小时/分钟的时间段事件（如 "9:00 - 10:30 设计评审"），但数据模型不支持。

   **建议**：
   - 方案 A：将 `start_date` 和 `due_date` 从 DATE 改为 DATETIME，支持精确时间
   - 方案 B：新增 `start_time` VARCHAR 和 `end_time` VARCHAR 字段，存储 "HH:MM" 格式
   - 方案 C：MVP 阶段周/日视图只显示全天事件，不支持精确时间段

   推荐方案 A，对现有模型改动最小，且与 SQLite 的 datetime 函数兼容。

3. **事件创建弹窗中的"所属项目"**：如果是在项目内的日历视图，项目已确定；如果是全局日历视图，需要选择项目。建议区分这两种场景，项目内日历隐藏项目选择字段。

4. **迷你日历的位置**：文档说"位置：侧边栏底部（标签区域下方）"。但侧边栏已经包含搜索、收藏夹、项目列表、标签、设置等内容，底部再加迷你日历会导致侧边栏需要滚动。建议迷你日历仅在日历视图激活时显示，替代侧边栏中的标签区域，或作为日历视图工具栏的一部分。

5. **月视图中的"跨周事件"渲染**：跨越周末到下周一的事件需要在两行中分别渲染，左侧平头 + 右侧圆角。这是日历视图中最复杂的布局计算之一。Schedule-X 是否内置支持此功能需要验证。

## 3. 跨视图一致性检查

### 3.1 任务详情面板

| 检查项 | 一致性 | 问题 |
|--------|--------|------|
| 面板宽度 | 不一致 | 项目列表视图定义为 400px，PRD overview 定义为 480px，看板视图引用 480px |
| 滑入动画 | 一致 | 从右向左滑入 |
| 关闭方式 | 一致 | Esc / 关闭按钮 / 点击外部 |
| 面板内容 | 一致 | 标题、属性、描述、子任务、删除 |
| 遮罩 | 一致 | 无遮罩（可同时操作左侧内容） |

**建议**：统一任务详情面板宽度为 480px（PRD overview 和看板视图的定义），项目列表视图的 400px 应更正。

### 3.2 拖拽交互

| 检查项 | 一致性 | 问题 |
|--------|--------|------|
| 拖拽库 | 不一致但合理 | 看板使用 dnd-kit，甘特图使用原生 pointer 事件，日历使用 Schedule-X 内置拖拽 |
| 视觉反馈 | 基本一致 | 都使用半透明原位占位 + 跟随鼠标的副本 |
| 释放动画 | 一致 | 200ms ease |
| 拖拽激活距离 | 一致 | 5px |

**建议**：三套独立的拖拽系统虽然增加了维护成本，但考虑到三种场景（DOM 列表、SVG 图形、日历网格）的差异性，分别实现是合理的。建议封装统一的视觉反馈工具函数（半透明度、阴影参数等）。

### 3.3 筛选/搜索

| 检查项 | 一致性 | 问题 |
|--------|--------|------|
| 搜索防抖 | 一致 | 300ms |
| 搜索范围 | 基本一致 | 项目列表：名称+描述；看板/列表：任务标题 |
| 筛选逻辑 | 一致 | AND 组合 |
| 筛选字段 | 基本一致 | 状态、优先级、标签、截止日期 |
| 筛选效果 | 不一致 | 看板：不匹配卡片 opacity 0.3；列表：隐藏不匹配行；日历：隐藏不匹配事件 |

**建议**：筛选效果的不一致是合理的（看板保持全局视图、列表追求信息密度），无需统一。但建议在列表视图中也提供"淡化模式"的选项，与看板保持一致的筛选体验。

## 5. 状态管理设计建议

### 5.1 全局状态（Zustand Stores）

| Store | 作用域 | 主要字段 |
|-------|--------|---------|
| `useProjectStore` | 全局 | projects, activeProjectId, loading |
| `useTaskStore` | 全局 | tasks（按项目缓存）, selectedTaskId |
| `useTagStore` | 全局 | tags |
| `useUIStore` | 全局 | sidebarCollapsed, theme, activeView |
| `useSettingsStore` | 全局 | 持久化的用户设置 |

### 5.2 视图本地状态

| 状态 | 存储方式 | 说明 |
|------|---------|------|
| 看板列折叠状态 | `useUIStore` + persist | 跨会话保持 |
| 看板分组方式 | URL 搜索参数 | 可分享/恢复 |
| 列表排序/筛选 | URL 搜索参数 | 可分享/恢复 |
| 列表列显隐 | `useSettingsStore` | 跨会话保持 |
| 甘特图缩放级别 | 组件 local state | 无需持久化 |
| 甘特图分割线位置 | `useSettingsStore` | 跨会话保持 |
| 日历当前视图/日期 | URL 搜索参数 | 可分享/恢复 |
| 拖拽中间状态 | 组件 local state / ref | 高频更新不走 Store |

### 5.3 数据查询策略建议

```
用户操作 → 调用 Service 层
                ↓
         Tauri IPC → Rust Command → SQLite 查询
                ↓
         返回数据 → 更新 Zustand Store
                ↓
         React 组件自动重渲染
```

**关键原则**：
- Store 中缓存的是"当前项目"的任务数据，切换项目时清空并重新加载
- 写操作（CRUD）采用乐观更新：先更新 Store，再异步写 SQLite
- 乐观更新失败时回滚 Store 并显示错误 Toast
- 视图切换不重新查询数据（共享同一个 Store），仅改变渲染方式

## 6. 风险清单与缓解措施

### 6.1 高风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 自研 SVG 甘特图工作量超预期 | 延期 2-4 周 | MVP 阶段裁剪依赖关系和关键路径功能，先实现基础时间轴 + 任务条 + 拖拽 |
| SVG 虚拟滚动实现复杂度高 | 性能不达标或开发时间长 | 使用自定义的 scroll-based 渲染裁剪替代 TanStack Virtual |
| 甘特图左右面板同步滚动抖动 | 用户体验差 | 使用单一外层滚动容器方案 |

### 6.2 中风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| Schedule-X 定制深度不足 | 需要 fork 或换库 | 开发前做 1-2 天 PoC 验证 |
| 日历视图数据模型缺少时间字段 | 周/日视图功能受限 | 将 DATE 改为 DATETIME |
| 看板大量卡片筛选时的 DOM 性能 | 500+ 卡片时卡顿 | 超过 200 卡片时隐藏不匹配卡片 |
| 列表视图列固定 + 虚拟滚动兼容性 | CSS sticky 在虚拟化容器中异常 | 提前做 PoC 验证 |

### 6.3 低风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 项目列表展开任务的虚拟滚动复杂 | 实现困难 | 限制展开显示数量 + "查看全部"链接 |
| 分组头拖拽在虚拟滚动中的表现 | 视觉异常 | MVP 不支持分组头拖拽 |
| 日历迷你导航占用侧边栏空间 | 侧边栏过长需滚动 | 仅日历视图激活时显示 |

## 7. MVP 裁剪建议

鉴于整体工作量预估 51-68 人天，如果需要缩短到 MVP 首版约 35-40 人天，建议以下裁剪：

### 7.1 Phase 1 - 核心 MVP（约 35 人天）

| 视图 | 裁剪内容 | 保留内容 | 工作量 |
|------|---------|---------|--------|
| 项目列表 | 卡片模式暂不实现 | 行模式、展开任务、新建项目 | 4 天 |
| 看板 | 按标签分组暂不实现、WIP 限制暂不实现 | 按状态/优先级分组、拖拽、筛选搜索 | 8 天 |
| 列表 | 分组头拖拽排序暂不实现 | 虚拟滚动、行内编辑、批量操作、分组、筛选 | 7 天 |
| 甘特图 | 依赖关系、关键路径、里程碑暂不实现 | 基础时间轴 + 任务条 + 拖拽调整日期 + 左右面板 | 12 天 |
| 日历 | 迷你日历暂不实现、拖拽调整时间暂不实现 | 月/周/日视图、事件创建弹窗、点击打开详情 | 8 天 |
| 共享组件 | - | TaskDetailPanel、筛选器、右键菜单等 | 5 天（分摊到各视图中） |

### 7.2 Phase 2 - 功能完善（约 20 人天）

- 甘特图依赖关系连线 + 创建/删除依赖（6 天）
- 甘特图关键路径计算和高亮（3 天）
- 甘特图里程碑（2 天）
- 日历拖拽交互（3 天）
- 日历迷你导航（2 天）
- 看板 WIP 限制 + 按标签分组（2 天）
- 项目列表卡片模式（2 天）

### 7.3 开发优先级建议

```
1. 共享组件（TaskDetailPanel、StatusBadge 等）
2. 项目列表视图（最简单，建立基础架构）
3. 列表视图（TanStack Table 基础设施，后续看板/甘特图的任务列表可复用）
4. 看板视图（核心交互，用户最常用）
5. 日历视图（Schedule-X 集成）
6. 甘特图视图（最复杂，放在最后以积累经验）
```
